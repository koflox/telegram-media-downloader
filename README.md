# tg-media-downloader

Консольная выгрузка **медиа из всех чатов** Telegram под вашим аккаунтом через
MTProto (Telethon). Качает:

- **фото** и **видео**;
- **фото и видео, отправленные файлом** (документом — `image/*`, `video/*`);
- **видео-кружочки** (round video / video note).

Стикеры, гифки, голосовые и аудио — пропускаются (настройка типов в `config.yaml`).

Рассчитан на **большие объёмы**: выгрузка резюмируемая, уже скачанные файлы
повторно не качаются.

---

## Установка

Нужен Python 3.10+.

```bash
python -m venv .venv
.venv\Scripts\activate          # Windows (PowerShell/cmd)
# source .venv/bin/activate     # Linux/macOS
pip install -r requirements.txt
```

## Настройка

`api_id`/`api_hash` уже прописаны в `config.yaml`.
Свои можно получить на <https://my.telegram.org> → **API development tools**.

Основные настройки `config.yaml`:

```yaml
download:
  output_dir: "./downloads"   # корневая папка; внутри подпапка на каждый чат
  photos: true                # обычные фото
  videos: true                # обычные видео
  video_notes: true           # кружочки
  file_photos: true           # фото, отправленные файлом
  file_videos: true           # видео, отправленные файлом
  max_file_mb: 0              # не качать файлы крупнее N МБ (0 = без лимита)
  since_date:                 # не качать сообщения старше даты (пусто = вся история)

scope:
  include_groups: true        # выгружать из групп/супергрупп
  include_channels: false     # выгружать из каналов
  exclude_chats: []           # id или @username чатов-исключений
```

Личные чаты выгружаются всегда.

## Запуск

```bash
python download.py                 # выгрузка из всех чатов (резюмируемая)
python download.py --rescan        # подхватить новые сообщения в уже пройденных чатах
python download.py --reset-progress # начать заново (сбросить прогресс)
```

При первом запуске Telethon попросит номер телефона и код из Telegram —
создастся файл сессии `media_downloader.session` (не коммитим).

## Куда складывается

```
downloads/
  <Название чата>_<chat_id>/
    <msg_id>_photo.jpg
    <msg_id>_video.mp4
    <msg_id>_round.mp4
    <msg_id>_photo_file.png
    <msg_id>_video_file.mkv
```

Имя файла детерминированное (`<id сообщения>_<тип>`), поэтому повторный запуск не
создаёт дублей.

## Пауза и продолжение

Прервать можно в любой момент (**Ctrl+C**) — прогресс сохраняется, повторный
запуск продолжит с места остановки. Резюмируемость работает на двух уровнях:

1. **Прогресс по чату** (`progress.json`): для каждого чата хранится id самого
   старого уже обработанного сообщения. На следующем запуске чат продолжается с
   этого места, а полностью выгруженные чаты пропускаются целиком — историю не
   приходится перебирать заново.
2. **Проверка файла на диске**: перед скачиванием проверяется, есть ли уже файл
   по нужному пути. Даже если `progress.json` потерян или удалён — уже скачанные
   файлы не качаются повторно. Закачка идёт во временный `*.part` и переименовывается
   только после полной загрузки, так что оборванный файл докачается на следующем
   запуске.

`--rescan` — пройти все чаты заново с самых новых сообщений (чтобы забрать то,
что появилось после прошлой выгрузки); уже скачанные файлы при этом пропускаются.

## Скорость

Искусственно скорость не режется (`pause_between_files: 0` по умолчанию). Для
ускорения сделано:

- **Параллельная загрузка (пул воркеров)** — `download_workers` постоянных
  воркеров (по умолчанию `4`) разбирают общую очередь заданий: producer читает
  сообщения и кладёт медиа в очередь, а освободившийся воркер сразу берёт
  следующее задание — **без ожидания «волны»** (никто не простаивает, пока
  доканчивается самый медленный файл). Главный выигрыш на массе мелких файлов
  (фото, кружочки). Это `asyncio`-конкурентность (один поток, кооперативная
  многозадачность), а не ОС-потоки — для сетевых загрузок это и нужно. Больше
  воркеров = быстрее, но выше риск `FloodWait` — разумно `4–8`. Все воркеры идут
  через одно соединение к дата-центру, поэтому выше пропускной способности этого
  канала (и скорости серверов Telegram) не прыгнуть.
  Прогресс резюма (`oldest_id`) сдвигается только по непрерывному завершённому
  префиксу сообщений, поэтому при `Ctrl+C`/обрыве недокачанные файлы не теряются —
  на следующем запуске они перечитываются и докачиваются.
- **cryptg** — C-реализация AES-IGE (в `requirements.txt`). MTProto шифрует весь
  трафик; без `cryptg` Telethon расшифровывает скачанное на чистом Python, и на
  больших объёмах это упирается в CPU. Telethon подхватывает `cryptg` сам, просто
  по факту установки.

Качество не страдает: параллелизм и `cryptg` влияют только на скорость, файлы
по-прежнему берутся в полном размере без пережатия.

## Лимиты Telegram (FloodWait)

`FloodWait` — общий на аккаунт, а не на отдельный запрос. Поэтому при параллельной
загрузке используется **общая заслонка**: как только любой воркер получает
`FloodWait`, до момента возобновления притормаживают **все** воркеры (а не каждый
сам по себе). Это не даёт остальным долбить сервер во время «бана» и продлевать
наказание. Берётся максимальный из полученных интервалов плюс `flood_buffer_seconds`.

Паузы между чатами и волнами файлов настраиваются в секции `throttle` конфига —
при очень больших объёмах и высоком `download_workers` имеет смысл выставить
`pause_between_files` (пауза между волнами), чтобы реже упираться в лимиты.

## Структура

```
download.py            точка входа (python download.py)
config.yaml            все настройки
tgmedia/
  config.py              разбор и валидация config.yaml
  scoping.py             классификация чатов (личка/группа/канал)
  downloader.py          работа с Telegram: scope + выгрузка медиа
  progress.py            резюмируемость (progress.json)
  logging_setup.py       настройка логов
  app.py                 сборка и основной сценарий
downloads/             скачанные медиа (создаётся при запуске)
```
